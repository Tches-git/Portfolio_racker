# 金融研究终端求职演示手册

这份手册用于大模型 / Agent 岗位面试演示。目标不是把每个功能都点一遍，而是在 3 到 5 分钟内讲清楚：这个项目如何把金融消息、RAG、多 Agent、评测和交付闭环串起来。

## 1. 演示主线

一句话定位：

> 这是一个面向金融研究场景的多用户 Agent 工作台，核心链路是“创建组合 → 实时采集 → 多 Agent 研报生成 → 引用审计 → 报告交付”。

推荐演示顺序：

1. 打开 `/`，展示空账号或风险驾驶舱。
2. 进入 `/watchlist`，说明用户主动创建组合，系统不会污染新用户数据。
3. 刷新组合事件，说明实时采集和事件结构化。
4. 进入 `/events`，展示事件 / 预警如何进入研究流程。
5. 进入 `/runs`，展示 Agent 任务流水。
6. 进入 `/runs/[runId]`，重点讲六角色 Agent Trace。
7. 进入 `/quality`，展示评测、引用可信度和工程门禁。

## 2. 面试讲法

### 风险驾驶舱

讲法：

> 首页不是普通 dashboard，而是研究入口。它把组合风险、事件流、待处理预警和研报任务放到一个工作路径里，帮助研究员决定下一步先处理什么。

强调点：

- 新用户默认空态，不加载演示数据。
- 数据只来自用户创建的组合。
- 事件和预警是研报任务的上游信号。

### 组合监控

讲法：

> 组合是用户的数据边界，也是事件采集边界。每个用户维护自己的股票池，刷新后事件、预警和研报都写入当前用户空间。

强调点：

- 多用户隔离。
- 股票名称 / 代码搜索。
- 手动刷新避免隐式污染。

### 事件预警台

讲法：

> 这里把金融消息从“新闻列表”变成结构化研究信号，包括事件类型、影响等级、情绪、置信度和来源证据。高影响或风险事件可以进一步转成 Agent 任务。

强调点：

- 事件去重、分类和影响评估。
- 预警规则命中。
- 事件详情像新闻页面，保留来源和处理动作。

### Agent 任务台

讲法：

> 任务台不是单纯的任务列表，而是多智能体研报生产流水线。每条任务都能看到当前角色、工具调用、引用覆盖和导出状态。

强调点：

- 任务状态可追踪。
- 失败可重试、可归档。
- 所有任务按当前用户隔离。

### 多 Agent 执行台

讲法：

> 我没有把研报生成做成一个黑箱 prompt，而是拆成 Planner、MarketData、EventAnalysis、RiskReview、ReportWriter、CitationAudit 六个角色，并把每个角色的输入、输出、耗时、工具调用和降级状态持久化到 Trace。

六角色说明：

- Planner：拆解研究目标与问题。
- MarketDataAgent：行情、财务、估值和趋势分析。
- EventAnalysisAgent：公告、新闻和触发事件解读。
- RiskReviewAgent：风险、数据缺口和异常复核。
- ReportWriterAgent：汇总前序角色形成写作 brief。
- CitationAuditAgent：检查引用覆盖和无来源观点。

### 质量指标页

讲法：

> 质量页不是为了堆数字，而是为了让简历上的指标有来源。事件评测、Agent 任务评测、公共金融 QA、RAG 引用审计和构建测试都可以复现。

强调点：

- Agent 任务评测：工具覆盖、Trace 完整率、角色覆盖。
- RAG 可信度：引用覆盖、来源数量、无来源观点。
- 公共金融 QA：避免只用项目自造样本。
- 工程门禁：pytest、typecheck、build。

## 3. 推荐回答面试追问

### 为什么不用单 Agent？

回答：

> 金融研报任务天然包含规划、数据分析、事件解读、风险复核、写作和审计几个阶段。单 Agent 容易把这些过程混在一个 prompt 里，难以观察和追责。多 Agent 的收益不是“更玄”，而是让角色边界、工具调用和错误降级更可控。

### 多 Agent 是否只是包装？

回答：

> 我做了角色级 Trace：每个角色有状态、摘要、工具调用数、耗时、降级和错误字段，并且进入任务详情页和运行指标。也就是说，页面能展示每个角色实际产出了什么，而不是只在文案里说用了多 Agent。

### 如何减少幻觉？

回答：

> 我从两个方向约束：一是 RAG 来源进入上下文，二是 CitationAuditAgent 对最终报告做引用覆盖和无来源观点统计。任务详情和质量页都会展示这些指标。

### 数据质量怎么保证？

回答：

> 事件层有去重、分类、影响评估和预警命中评测；研报层有 RAG 引用审计；Agent 层有任务评测和 Trace 完整率。指标只写项目实际跑出来的结果，不预设夸张业务效果。

### 如果数据源失败怎么办？

回答：

> 页面和 Agent Trace 都会显示降级状态，例如使用缓存、缺失字段或角色降级。这样演示和真实使用时不会静默失败。

## 4. 演示前检查清单

- 当前账号是否为空态，或是否已有一组可演示组合。
- API 是否运行：`http://127.0.0.1:8000/api/v1/health`
- 前端是否运行：`http://localhost:3004`
- `/runs` 是否有至少一个完成任务用于展示 Trace。
- `/quality` 是否有最近一次评测结果。
- 报告导出物是否能点“预览”打开。

## 5. 简历口径

可写：

> 基于 AutoGen 设计多角色 Agent 研报生成链路，将金融研究任务拆分为规划、行情与财务分析、事件解读、风险复核、报告写作和引用审计等阶段，并持久化角色级 Trace、工具调用、降级状态和引用可信度指标，提升复杂研报生成过程的可解释性与可复现性。

不要写：

- 不要写“完全自动替代分析师”。
- 不要写无法复现的夸张收益率。
- 不要只写“新增 RAG 引用可信度评测”这种版本更新式表达。
- 不要把项目描述成单 ReAct Agent。
